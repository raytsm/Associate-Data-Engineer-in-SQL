\section{Data Warehouse Basics}
\subsection{What is data warehouse?}
A computer system designed to store and analyze large amount of data for an organization.

\subsubsection{What does a data warehouse do?}
\begin{itemize}
    \item Gathers data from different areas of an organization
    \item Integrates and stores the data
    \item Make it available for analysis
\end{itemize}

\subsubsection{Why is a data warehouse valuable?}
Organizatons implement data warehouses in order to:*
\begin{itemize}
    \item Support business intelligence activity
    \item Enable effective organizational analysis and decision-making
    \item Find ways to innovate based on insights from their data
\end{itemize}

\subsection{What's the difference between data warehouses and data lakes?}
\subsubsection{Database}
\begin{figure}[H]
    \centering
    \caption{Databases}
    \includegraphics[width=0.5\textwidth]{figures/Databases.png}
\end{figure}
\begin{itemize}
    \item Structured data in rows and columns
    \item Transactional databases store transactions
\end{itemize}

\subsubsection{Data warehouse}
\begin{figure}[H]
    \centering
    \caption{Data Warehouse}
    \includegraphics[width=0.5\textwidth]{figures/DataWarehouse.png}
\end{figure}
\begin{itemize}
    \item Gather data, integrate, and make available for analysis
    \item Many input data sources
    \item Stores structured data
    \item Complex to change
    \begin{itemize}
        \item Upstream and downstream effects must be considered
    \end{itemize}
    \item Typically >100 GB in size
\end{itemize}

\subsubsection{Why the data warehouse?}
\begin{itemize}
    \item How quickly the query will run on a large amount of data
    \item Avoid slowing down transactional database
\end{itemize}

\subsubsection{Data marts}
\begin{figure}[H]
    \centering
    \caption{Data marts}
    \includegraphics[width=0.5\textwidth]{figures/DataMart.png}
\end{figure}
\begin{itemize}
    \item A relational databases for analysis
    \item Data is focused on one subject area
    \item Few input data sources
    \item Typically less than 100 GB in size
\end{itemize}

\subsubsection{Data lake}
\begin{figure}[H]
    \centering
    \caption{Data Lake}
    \includegraphics[width=0.5\textwidth]{figures/DataLake.png}
\end{figure}
\begin{itemize}
    \item Entire organization store of data
    \begin{itemize}
        \item Contains data from many departments
        \item Many data input sources
        \item Typically more than 100 GB in size
    \end{itemize}
    \item Stores structured and Unstructured data
    \begin{itemize}
        \item Examples: video, audio, and documents
    \end{itemize}
    \item Less complex to make Changes
    \begin{itemize}
        \item Fewer upstream and downstream effects to consider
    \end{itemize}
    \item Purpose to store data may not be known thus Less organized
\end{itemize}

\subsection{Data warehouse Architectures}
\subsubsection{Inmon by Bill Inmon - top-down approach}
\begin{figure}[H]
    \centering
    \caption{Top down approach}
    \includegraphics[width=1.0\textwidth]{figures/Inmon.png}
\end{figure}
Must decide:
\begin{itemize}
    \item On all data definitions, cleaning, and business rules
    \item Before any data enters warehouse
\end{itemize}

\subsubsection{Pros and cons of top-down}
\textbf{Advantages}
\begin{itemize}
    \item Single source of truth for organization
    \item Normalization = less storage
    \item Easy to change data marts to support reporting changes
\end{itemize}

\textbf{Disadvantages}
\begin{itemize}
    \item More joins = slower response time
    \item Lengthy upfront Work causing Higher startup cost
\end{itemize}

\subsubsection{Kimball by Ralph Kimball - bottom-up approach}
\begin{figure}[H]
    \centering
    \caption{Bottom Up approach}
    \includegraphics[width=1.0\textwidth]{figures/Kimball.png}
\end{figure}
\begin{itemize}
    \item Denormalizes data
    \item Focus on departmental data mart
    \item Data moves directly from ETL to data marts
\end{itemize}

\subsubsection{Pros and cons of bottom-up}
\textbf{Advantages:}
\begin{itemize}
    \item Upfront development speed causing Lower startup cost
    \item Denormalized = user friendly
\end{itemize}

\textbf{Disadvantages:}
\begin{itemize}
    \item Increased ETL processing time
    \item Greater possibility of duplicate data
    \item Ongoing development needed
\end{itemize}

\subsection{OLAP and OLTP systems}
\subsubsection{OLAP systems}
\begin{itemize}
    \item OLAP(online analytical processsing)
    \item Designed to support analysis of large amounts of data
    \item Example dimensional organization
    \begin{itemize}
        \item country, state, city
        \item years, months, days
    \end{itemize}
    \item OLAP reorganizes data into multidimensional format
\end{itemize}

\subsubsection{OLAP cube}
\begin{figure}[H]
    \centering
    \caption{OLAP cube}
    \includegraphics[width=0.8\textwidth]{figures/OLAP cube.png}
\end{figure}
\begin{itemize}
    \item OLAP cube key to OLAP system
    \item Faster processing vs. traditional relational databases
    \item Hypercubes have more than three dimensions
\end{itemize}

\subsubsection{OLTP}
\begin{itemize}
    \item OLTP(online transaction processing)
    \item Designed for processing simple database queries
    \item Used in source systems to data warehouse
\end{itemize}

\subsubsection{Example for a credit card company}
\textbf{OLTP}:
\begin{itemize}
    \item System tracks customer's purchase
    \item Processes large amounts of simple database updates to account balances
\end{itemize}

\textbf{OLAP}:
\begin{itemize}
    \item Designed for analyzing purchase data
    \item Data organized by multiple dimensions
\end{itemize}
