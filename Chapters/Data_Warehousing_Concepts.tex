\section{Data Warehouse Basics}
\subsection{What is data warehouse?}
A computer system designed to store and analyze large amount of data for an organization.

\subsubsection{What does a data warehouse do?}
\begin{itemize}
    \item Gathers data from different areas of an organization
    \item Integrates and stores the data
    \item Make it available for analysis
\end{itemize}

\subsubsection{Why is a data warehouse valuable?}
Organizatons implement data warehouses in order to:*
\begin{itemize}
    \item Support business intelligence activity
    \item Enable effective organizational analysis and decision-making
    \item Find ways to innovate based on insights from their data
\end{itemize}

\subsubsection{Pros and cons of top-down}
\textbf{Advantages}
\begin{itemize}
    \item Single source of truth for organization
    \item Normalization = less storage
    \item Easy to change data marts to support reporting changes
\end{itemize}

\textbf{Disadvantages}
\begin{itemize}
    \item More joins = slower response time
    \item Lengthy upfront Work causing Higher startup cost
\end{itemize}

\subsubsection{Kimball by Ralph Kimball - bottom-up approach}
\begin{figure}[H]
    \centering
    \caption{Bottom Up approach}
    \includegraphics[width=1.0\textwidth]{figures/Kimball.png}
\end{figure}
\begin{itemize}
    \item Denormalizes data
    \item Focus on departmental data mart
    \item Data moves directly from ETL to data marts
\end{itemize}

\subsubsection{Pros and cons of bottom-up}
\textbf{Advantages:}
\begin{itemize}
    \item Upfront development speed causing Lower startup cost
    \item Denormalized = user friendly
\end{itemize}

\textbf{Disadvantages:}
\begin{itemize}
    \item Increased ETL processing time
    \item Greater possibility of duplicate data
    \item Ongoing development needed
\end{itemize}

\subsection{Kimball's four step process}
\begin{enumerate}
    \item \textbf{Step 1 - Select the organizational process}
    \begin{itemize}
        \item Ask questions about a process
        \item Kimball bottom-up approach starts with a business process \\
        \textbf{Examples of organizationl process}
        \begin{itemize}
            \item Invoice and billing
            \item Product quality monitoring
            \item Marketing
        \end{itemize}
    \end{itemize}
    \item \textbf{Step 2 - Declare the grain}
    \begin{itemize}
        \item Grain = level to store fact table
        \item A level of data that cannot be split further \\
        \textbf{Examples of business process}
        \begin{itemize}
            \item Music service can have Song grain
            \item Shipping service can have Line item grain
        \end{itemize}
    \end{itemize}
    \item \textbf{Step 3 - Identify the dimensions}
    \begin{itemize}
        \item Choose dimensions that apply to each row 
        \item How to describe the data? 
        \item Business users and analysts = valuable feedback \\
        \textbf{Examples of common dimensions:}
        \begin{itemize}
            \item \textbf{Time:} year, quarter, and month
            \item \textbf{Location:} address, state, and country
            \item \textbf{Users:} names and email address
        \end{itemize}
    \end{itemize}
    \item \textbf{Step 4 - Identify the facts}
    \begin{itemize}
        \item Numerical facts for each fact table row
        \item What are we answering?
        \item Metrics should be true at selected grain \\
        \textbf{Examples of facts:}
        \begin{itemize}
            \item Music service: total number of plays, sales revenue of a song
            \item Ride-sharing: travel distance, time needed
        \end{itemize}
    \end{itemize}
\end{enumerate}

\subsection{Slowly changing dimensions}
Kimball's approach to modeling slowly changing dimensions.
\subsubsection{The challenge}
Initially we had electric-veh. in category column but now we want it to be electric-crossover. There are 3 ways of doing 
this by kimball's approach. Type I, Type II and Type III. It focus on the dimension and not the fact table.

\textbf{Original table} \\
\begin{tabular}{lll}
    \textbf{ProductID} & \textbf{Description} & \textbf{Category} \\
    12345 & Tesla-ModelY & electric-veh.
\end{tabular}

Update Category:
\begin{itemize}
    \item \textbf{Current:} electric-veh.
    \item \textbf{New:} electric-crossover
\end{itemize}

\subsubsection{Type I:}
\begin{itemize}
    \item Update value in table
    \item Will lose any history
\end{itemize}

\textbf{Original table} \\
\begin{tabular}{lll}
    \textbf{ProductID} & \textbf{Description} & \textbf{Category} \\
    12345 & Tesla-ModelY & \st{electric-veh.}
\end{tabular}

\textbf{New table} \\
\begin{tabular}{lll}
    \textbf{ProductID} & \textbf{Description} & \textbf{Category} \\
    12345 & Tesla-ModelY & electric-crossover
\end{tabular}

\subsubsection{Type II:}
\begin{itemize}
    \item Add a row with the updated value
    \item The history is retained
\end{itemize}

\textbf{Original table} \\
\begin{tabular}{lll}
    \textbf{ProductID} & \textbf{Description} & \textbf{Category} \\
    12345 & Tesla-ModelY & electric-veh.
\end{tabular}

\textbf{New table} \\
\begin{tabular}{lllll}
    \textbf{ProductID} & \textbf{Description} & \textbf{Category} & \textbf{StartDate} & \textbf{EndDate} \\
    12345 & Tesla-ModelY & electric-veh. & 1970-01-01 & 2022-03-10 \\
    20053 & Tesla-ModelY & electric-crossover & 2022-03-11 & 2050-12-31 
\end{tabular}

Here, the table consist of new row for the updated value including the date and new ID.

\subsubsection{Type III:}
\begin{itemize}
    \item Add column to dimension table to track changes.
    \item Can view past and current data together
    \item Can require reporting changes and limited tracking
\end{itemize}

\textbf{Original table} \\
\begin{tabular}{lll}
    \textbf{ProductID} & \textbf{Description} & \textbf{Category} \\
    12345 & Tesla-ModelY & electric-veh.
\end{tabular}

\textbf{New table} \\
\begin{tabular}{llll}
    \textbf{ProductID} & \textbf{Description} & \textbf{Category} & \textbf{PastCategory}\\
    12345 & Tesla-ModelY & electric-crossover & electric-veh.
\end{tabular}

\subsubsection{Modern approach}
\begin{itemize}
    \item Snapshot the whole dimension table
    \item Use historical snapshots for historical reports
    \item Similar to Type I but with historical recovery via snapshots.
\end{itemize}

\subsubsection{Basics of computer storage}

    \item Reading fewer blocks increases the overall speed of the process.

\subsubsection{Row store example}
\begin{itemize}
    \item Each row is stored in single block.
    \item But questions like what is the average hospitalization percentage across all age group for the 2019 season, 
    the system will have to read each block containing data for 2019 which in this example is 3 blocks to be read.
    \item Ideal for transactional workloads
\end{itemize}

    \includegraphics[width=1.0\textwidth]{figures/RSExample.png}

\subsubsection{Column store example}
\begin{itemize}
    \item It stores data for column in single block
    \item for the average hospitalization percentage for across all age group it only need to read 
    single block which is faster for transactional workloads but it's tradeoff is that adding data will take longer.
    \item It also has added benefits of better data compression because all of the data in a block is the same type.
    \item Ideal for analytical workloads.
\end{itemize}

\subsection{ETL and ELT}
\subsubsection{Understanding ETL and ELT names}
ETL : Extract, Transform and Load \\
ELT : Extract, Load and Transform

\subsubsection{Understanding ETL - Pros and Cons}
\begin{itemize}
    \item Data transformed during the move
    \item Uses separate system to process data
\end{itemize}

\textbf{Pros:}
\begin{itemize}
    \item Lower data storage cost
    \item PII security compliance
\end{itemize}

\textbf{Cons:}
\begin{itemize}
    \item Transformation errors/changes require new data pulls
    \item Costs of separate system to process data
\end{itemize}

\subsubsection{Understanding ELT - Pros and Cons}
\begin{itemize}
    \item Data is loaded, then transformed
    \item Uses the warehouse to transform the data
\end{itemize}

\textbf{Pros:}
\begin{itemize}
    \item No separate system to process data
    \item Transformations can be rerun without impacting source systems
    \item Works well for near real-time requirements
\end{itemize}

\textbf{Cons:}
\begin{itemize}
    \item Increased storage needs from raw data
    \item Compliance with PII security standards
\end{itemize}

    \item Sales data mart must be the priority

\subsubsection{Kimball - select the organizational process (step 1)}
\textbf{Considerations}
\begin{itemize}
    \item What type of customers purchase large volumes of photos?
\end{itemize}
\textbf{Decision:}
\begin{itemize}
    \item Develop customer purchases
\end{itemize}

\subsubsection{Kimball - Declare the grain (Step 2)}
\textbf{Considerations:}
\begin{itemize}
    \item Data should be flexible to answer many questions
    \item Selecting the lowest grain possible
\end{itemize}
\textbf{Decision}
\begin{itemize}
    \item Tracking customer/photo purchases
\end{itemize}

\subsubsection{Kimball - Identify the dimensions (Step 3)}
\textbf{Considerations:}
\begin{itemize}
    \item How do users describe the data that results from the business process?
    \item Customer prioritization
\end{itemize}
\textbf{Decision:}
\begin{itemize}
    \item Customer location (country and state)
    \item Date customer joined
    \item Default payment method
\end{itemize}

\subsubsection{Kimball -Identify the facts (Step 4)}
\textbf{Considerations:}
\begin{itemize}
    \item What are we answering?
\end{itemize}
\textbf{Decision:}
\begin{itemize}
    \item Time spent viewing photo before purchase
    \item Photo cost and tax
    \item Date of purchase
\end{itemize}

After these four step we now have star schema data model.
